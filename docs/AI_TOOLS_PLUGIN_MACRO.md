# AI Tools Plugin Macro Framework

Complete guide for exposing AI-accessible tools from plugins using Rust procedural macros.

## Overview

The AI Tools macro framework allows plugin developers to expose functions as AI-accessible tools with minimal boilerplate. You write your function, add a `#[ai_tool]` macro, point to documentation, and the framework handles everything else.

```rust
#[ai_tool(
    category = "refactoring",
    docs = "src/ai_tools/docs/rename_node.md"
)]
pub fn refactor_blueprint_rename_node(
    old_name: String,
    new_name: String,
) -> Result<Value, PluginError> {
    // Your implementation
}
```

The macro automatically generates:
- JSON schema for parameters
- Parameter extraction wrapper for JSON dispatch
- Documentation constant with embedded markdown
- Tool definition constant

```
┌─────────────────────────────────────────┐

            ▼

└─────────────────────────────────────────┘
```

## Quick Start

### 1. Create Your Function

```rust
use serde_json::Value;
use plugin_editor_api::PluginError;

pub fn my_tool(
    name: String,
    count: i32,
) -> Result<Value, PluginError> {
    // Your implementation here
    Ok(serde_json::json!({
        "status": "success",
        "processed": count
    }))
}
```

### 2. Add the Macro

```rust
#[ai_tool(
    category = "text-processing",
    docs = "src/ai_tools/docs/my_tool.md"
)]
pub fn my_tool(
    name: String,
    count: i32,
) -> Result<Value, PluginError> {
    // Your implementation
}
```

### 3. Create Documentation File

```markdown
# my_tool

Process text with the given parameters.

## Parameters

- **name** (string): The text to process
- **count** (i32): Number of times to process

## Returns

```json
{
    "status": "success",
    "processed": 5
}
```

## Best Practices

1. Always validate inputs
2. Return structured JSON
3. Use clear error messages
```

### 4. Register in Your Plugin

```rust
use plugin_ai_tools::ToolRegistry;
use lazy_static::lazy_static;

lazy_static! {
    static ref TOOL_REGISTRY: ToolRegistry = {
        let mut registry = ToolRegistry::new();
        
        registry.register(
            "my_tool",
            serde_json::json!({
                "name": "my_tool",
                "description": "Process text with parameters",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "name": {"type": "string"},
                        "count": {"type": "integer"}
                    },
                    "required": ["name", "count"]
                },
                "category": "text-processing"
            }),
            TOOL_DOC_MY_TOOL,                    // Generated by macro
            my_tool_ai_tool_wrapper,              // Generated by macro
        );
        
        registry
    };
}

impl EditorPlugin for MyPlugin {
    fn ai_tools(&self) -> Vec<AiToolDefinition> {
        TOOL_REGISTRY.definitions()
    }
    
    fn execute_ai_tool(
        &self,
        file_path: &Path,
        tool_name: &str,
        tool_args: Value,
    ) -> Result<Value, PluginError> {
        TOOL_REGISTRY.execute(tool_name, tool_args)
    }
}
```

## Macro Attributes

### `category` (optional)

Categorizes the tool for organization:

```rust
#[ai_tool(category = "refactoring")]
```

Common categories:
- `refactoring` - Rename, restructure, reformat
- `validation` - Check for errors or inconsistencies
- `analysis` - Analyze performance or quality
- `generation` - Create or generate content
- `optimization` - Improve performance or structure
- `search` - Find patterns or elements
- `navigation` - Move between elements

### `timeout_ms` (optional)

Execution timeout in milliseconds (default: 5000):

```rust
#[ai_tool(timeout_ms = 10000)]
```

### `docs` (optional)

Path to markdown documentation file:

```rust
#[ai_tool(docs = "src/ai_tools/docs/my_tool.md")]
```

The file path is relative to the crate root. Missing files cause compilation error.

## Generated Code

For a function like:

```rust
#[ai_tool(category = "refactoring", docs = "src/docs/rename.md")]
pub fn rename_item(old_name: String, new_name: String) -> Result<Value, PluginError> {
    // ...
}
```

The macro generates:

### Tool Definition Constant

```rust
pub const TOOL_DEF_RENAME_ITEM: &str = r#"{
    "name": "rename_item",
    "description": "Rename a item",
    "parameters": {
        "type": "object",
        "properties": {
            "old_name": {"type": "string"},
            "new_name": {"type": "string"}
        },
        "required": ["old_name", "new_name"]
    },
    "category": "refactoring"
}"#;
```

### Parameter Wrapper Function

```rust
pub fn rename_item_ai_tool_wrapper(
    tool_args: serde_json::Value,
) -> Result<serde_json::Value, PluginError> {
    let old_name: String = tool_args
        .get("old_name")
        .and_then(|v| v.as_str())
        .ok_or_else(|| PluginError::InvalidParameter("old_name".to_string()))?
        .to_string();
    
    let new_name: String = tool_args
        .get("new_name")
        .and_then(|v| v.as_str())
        .ok_or_else(|| PluginError::InvalidParameter("new_name".to_string()))?
        .to_string();
    
    rename_item(old_name, new_name)
}
```

### Documentation Constant

```rust
pub const TOOL_DOC_RENAME_ITEM: &str = include_str!("src/docs/rename.md");
```

This embeds the entire markdown file at compile time. If the file doesn't exist, compilation fails with a clear error.

## Parameter Types

The macro supports standard Rust types and automatically generates JSON schema:

| Rust Type | JSON Type | Example |
|-----------|-----------|---------|
| `String` | `string` | `"hello"` |
| `bool` | `boolean` | `true` |
| `i32`, `i64` | `integer` | `42` |
| `u32`, `u64` | `integer` | `100` |
| `f32`, `f64` | `number` | `3.14` |
| `Option<T>` | Optional | `null` or value |

For `Option<T>` types, the parameter is automatically marked as not required.

## Error Handling

### Macro-Generated Wrapper

The wrapper function handles parameter extraction errors:

```rust
pub fn my_tool_ai_tool_wrapper(
    tool_args: Value,
) -> Result<Value, PluginError> {
    // Returns PluginError if parameter missing or wrong type
    let param: String = extract_param(tool_args, "param")?;
    my_tool(param)
}
```

### Your Function

Return `Result<Value, PluginError>`:

```rust
#[ai_tool(docs = "docs/my_tool.md")]
pub fn my_tool(name: String) -> Result<Value, PluginError> {
    if name.is_empty() {
        return Err(PluginError::ValidationError("name cannot be empty".to_string()));
    }
    
    Ok(json!({"status": "success"}))
}
```

## Crates

### `crates/plugin_ai_macro`

Procedural macro crate for `#[ai_tool]` attribute macro.

**Location**: `crates/plugin_ai_macro/src/lib.rs`

**Key Features**:
- Parse `category`, `timeout_ms`, `docs` attributes
- Generate tool definition JSON schema
- Generate parameter extraction wrapper
- Embed markdown via `include_str!()`
- Compile-time validation of doc files

**Dependencies**:
- `proc-macro2`
- `quote`
- `syn`
- `plugin_editor_api`

### `crates/plugin_ai_tools`

Runtime support library for tool management.

**Location**: `crates/plugin_ai_tools/src/lib.rs`

**Key Types**:

#### `ToolRegistry`

```rust
pub struct ToolRegistry {
    tools: HashMap<String, RegisteredTool>,
}

impl ToolRegistry {
    pub fn new() -> Self { ... }
    pub fn register(
        &mut self,
        name: &str,
        definition: Value,
        docs: &str,
        handler: Box<dyn Fn(Value) -> Result<Value, PluginError>>,
    ) { ... }
    pub fn execute(
        &self,
        tool_name: &str,
        args: Value,
    ) -> Result<Value, PluginError> { ... }
    pub fn definitions(&self) -> Vec<AiToolDefinition> { ... }
    pub fn documentation(&self, tool_name: &str) -> Option<String> { ... }
    pub fn tool_names(&self) -> Vec<String> { ... }
}
```

#### `AiToolBuilder`

Fluent interface for easy tool definition:

```rust
let tool = AiToolBuilder::new("my_tool", "Do something")
    .parameter("name", "string", "The name")
    .parameter("count", "integer", "How many times")
    .optional_parameter("verbose", "boolean", "Print details")
    .category("processing")
    .build();
```

#### `AiToolProvider` Trait

For plugins to implement:

```rust
pub trait AiToolProvider {
    fn create_tool_registry(&self) -> ToolRegistry;
    fn capabilities_for_file(&self, file_path: &Path) -> Vec<String>;
}
```

## Workflow

### Developer Perspective

```
1. Write function with logic
   ↓
2. Add #[ai_tool] macro with docs path
   ↓
3. Create markdown documentation file
   ↓
4. Build - macro generates constants and wrappers
   ↓
5. Register tools in EditorPlugin::ai_tools()
   ↓
6. Plugin loaded - tools discoverable to AI
```

### AI Perspective

```
1. Query available tools for file type
   ↓
2. System returns tool list with descriptions
   ↓
3. AI selects tool and asks for docs
   ↓
4. System returns full markdown documentation
   ↓
5. AI understands parameters and usage
   ↓
6. AI constructs JSON with parameters
   ↓
7. System executes tool via wrapper function
   ↓
8. Plugin function executes, returns result
   ↓
9. AI receives result and continues reasoning
```

## Performance

### Compile Time

- Macro expansion: One-time during build
- `include_str!()`: Compile-time file inclusion
- JSON schema generation: Compile-time via proc macro
- No runtime overhead for generation

### Runtime

- Tool lookup: O(1) HashMap access
- Parameter extraction: Single JSON traversal
- No allocations except for user data
- Documentation retrieval: Direct string reference

### Scalability

With macro-based framework:
- 10 plugins × 10 tools = 100 tools ✅
- 100 plugins × 100 tools = 10,000 tools ✅
- 1,000 plugins × 50 tools = 50,000 tools ✅

All work done at compile time. Runtime is just lookup and dispatch.

## FAQ

### Q: Can I have tools without documentation?

A: Yes, but not recommended. Omit the `docs` attribute:

```rust
#[ai_tool(category = "processing")]
pub fn my_tool(param: String) -> Result<Value, PluginError> { ... }
```

The macro generates an auto-generated documentation constant instead of using `include_str!()`.

### Q: What file types are supported for docs?

A: Markdown files (`.md`) via `include_str!()`. The framework doesn't process the content; it just embeds the text. You can write anything in the markdown file.

### Q: How do I handle complex return types?

A: Always return `Result<Value, PluginError>`:

```rust
#[ai_tool(docs = "docs/tool.md")]
pub fn my_tool() -> Result<Value, PluginError> {
    Ok(json!({
        "nested": {
            "structure": {
                "with": "any complexity"
            }
        },
        "arrays": [1, 2, 3],
        "mixed": ["string", 42, true, null]
    }))
}
```

### Q: Can I use custom types as parameters?

A: Only types that implement `FromJson`. Currently supported:
- `String`
- `bool`
- `i32`, `i64`, `u32`, `u64`
- `f32`, `f64`
- `Option<T>` where T is one of above

For complex types, accept `String` or `Value` and parse in your function.

### Q: How do I debug generated code?

A: Use `cargo expand` (via `cargo-expand`):

```bash
cargo install cargo-expand
cargo expand -p my_plugin_crate
```

Shows the expanded macro code.

### Q: What happens if the docs file is missing?

A: **Compilation fails** with clear error:

```
error: cannot find file `src/ai_tools/docs/my_tool.md`
  --> src/lib.rs:5:1
   |
 5 | #[ai_tool(docs = "src/ai_tools/docs/my_tool.md")]
   |
```

This is intentional - ensures all tools have documentation.

### Q: Can plugins have tools without the macro?

A: Yes, use `ToolRegistry::register()` directly:

```rust
let mut registry = ToolRegistry::new();
registry.register(
    "my_tool",
    json!({
        "name": "my_tool",
        "description": "Do something",
        "parameters": {...}
    }),
    "Tool documentation",
    Box::new(|args| {
        // Handle execution
    }),
);
```

But using the macro is strongly recommended for consistency and reduced boilerplate.
