# Blueprint System Architecture

Full end-to-end picture of how blueprints flow from the editor graph to compiled Rust code.

---

## The Four Stages

```
1. Editor Graph  →  2. JSON (graph_save.json)  →  3. Rust codegen (events.rs)  →  4. Bytecode (bytecode.json)
```

The JSON is **always** the source of truth. Rust is always regenerated; never edited by hand.

---

## Stage 1 — Editor Graph (Plugin_Blueprints)

The blueprint editor (`Plugin_Blueprints/src/editor/panel.rs`) lets users drag nodes onto a canvas and connect pins with wires.

The in-memory representation is `GraphDescription` from Graphy:
```rust
// Graphy/src/core/
GraphDescription {
    nodes: HashMap<NodeId, NodeInstance>,
    connections: Vec<Connection>,
}
```

Each `NodeInstance` has:
- `node_type: String` — matches the key in PBGC's node metadata registry
- `inputs: Vec<Pin>` — data + exec input pins with `connected_to` lists
- `outputs: Vec<Pin>` — exec output pins with `connected_to` lists
- `properties: HashMap<String, Value>` — literal values on unconnected pins

---

## Stage 2 — JSON Serialisation

On save (or compile), the graph is serialised to:
```
src/classes/<ClassName>/graph_save.json
```

Format:
```json
{
  "format_version": 1,
  "main_graph": {
    "nodes": { "<uuid>": { "node_type": "print_number", "inputs": [...], "outputs": [...] } },
    "connections": [ { "source_node": "...", "source_pin": "Body", "target_node": "...", "target_pin": "exec" } ]
  },
  "local_macros": [],
  "variables": []
}
```

---

## Stage 3 — Rust Codegen (PBGC)

`Plugin_Blueprints/src/features/compilation/compiler.rs` calls `BlueprintCompiler::compile()`.

PBGC pipeline:

```
graph_save.json
    │
    ▼
GraphDescription (Graphy types)
    │
    ▼
DataResolver.resolve_inputs()   ← figures out what value each input pin has
    │
    ▼
ExecutionRouting.build()        ← traces exec-pin connections into ordered call chains
    │
    ▼
BlueprintCodeGenerator.generate_event_fn("begin_play")
    │  For each node in exec order:
    │    generate_node_call() → "let node_<uuid>_result = fn_name(arg1, arg2);"
    │  For control-flow nodes:
    │    inline_control_flow_function() (Graphy AST transformer)
    │      → replaces exec_output!("Label") with the chain connected to that output
    │
    ▼
Rust function body string
    │
    ▼
gen_events_rs() in project.rs wraps it in:
    use pulsar_game::prelude::*;
    use pulsar_std::*;
    #[derive(EngineClass, Clone)]
    pub struct ClassName {}
    impl Actor for ClassName { fn begin_play(...) { logic::begin_play(); } }
    mod logic {
        use pulsar_std::*;
        pub fn begin_play() { /* generated body */ }
    }
    │
    ▼
Written to src/classes/<ClassName>/events/events.rs
```

### Control-flow node inlining (the tricky part)

Sequence node template (`PBGC/src/metadata.rs` `function_source`):
```rust
fn sequence() {
    exec_output!("Then0");
    exec_output!("Then1");
    exec_output!("Then2");
    exec_output!("Then3");
}
```

`inline_control_flow_function()` in `Graphy/src/utils/ast_transform.rs`:
- Parses the template with `syn`
- Replaces each `exec_output!("Then0")` stmt with a `Block { stmts: [...] }` containing **all** statements from the connected chain
- Unconnected outputs are replaced with empty blocks `{}` (never left as bare `exec_output!` — that would be a compile error)

## Stage 4 — Bytecode (pulsar_bp_executor)

`blueprint_compiler` also compiles the graph to `bytecode.json` (a flat instruction array):
```json
{
  "event_programs": {
    "begin_play": {
      "instructions": [
        { "Call": { "node_type": "print_number", "fn_ptr": 0, "input_offsets": [0] } },
        "Return"
      ]
    }
  }
}
```

The VM (`pulsar_bp_executor`) loads this at runtime and executes it. `fn_ptr: 0` is patched to the real function pointer during `BlueprintDispatcher::new()` → `BpExecutor::prepare()`.

## Generated Class File Layout

```
src/classes/
├── mod.rs                        ← regenerated by Build Core from disk scan
└── <ClassName>/
    ├── mod.rs                    ← pub mod vars; pub mod events; pub use events::*;
    ├── graph_save.json           ← source of truth for the editor
    ├── vars_save.json
    ├── vars/
    │   └── mod.rs               ← variable storage: Cell<f64>, RefCell<String>, etc.
    └── events/
        ├── mod.rs               ← pub mod events; pub use events::*;
        ├── events.rs            ← generated by PBGC; DO NOT EDIT
        └── .build/
            └── bytecode.json    ← generated bytecode; loaded by BlueprintDispatcher
```

## Adding a Control-Flow Node

1. Add `function_source` to the node metadata in `PBGC/src/metadata.rs`
2. Use `exec_output!("OutputName")` for each exec output pin
3. The AST transformer handles the rest automatically
4. Write regression tests in `Graphy/tests/ast_transform.rs` using `inline_control_flow_function` and `extract_exec_output_labels`

## Regression Tests for exec_output! substitution

`Graphy/tests/ast_transform.rs` has a full suite covering:
- Single/multiple outputs
- Multi-statement chains (the historical bug: `first_stmt.clone()` dropped all but first)
- Unconnected outputs (must produce `{}`, never bare `exec_output!`)
- Nested Sequence nodes
- Long chains (5+ nodes after a single output)

Always run `cargo test -p graphy` after touching `ast_transform.rs`.
